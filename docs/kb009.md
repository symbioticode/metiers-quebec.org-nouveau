# kb009 — Collision guard : analyse du conflit de préfixes

## Problème

Le `find_candidates_for_slug()` dans `graph_bridge.py` utilise un guard anti-collision pour éviter que les slugs frères (ex: `vendeur_autos`) ne soient traités comme des sous-concepts d'un autre slug (ex: `vendeur`).

Le guard `any(key.startswith(s+"_") for s in corpus_slugs if s != slug)` fonctionne pour **vendeur** (exclut correctement `vendeur_autos_*`) mais échoue pour **vendeur_autos** lui-même :

1. `vendeur_autos` n'a PAS de clé exacte dans `prefix_map` — les noeuds sont stockés sous `vendeur_autos_representant_en_vente_automobile`
2. Le prefix-scan pour `vendeur_autos` trouve `vendeur_autos_representant...`
3. Le guard exclut cette clé car elle commence par `vendeur_` (slug frère `vendeur`)
4. Résultat : `vendeur_autos` retourne 0 candidats → manquant

## Conflit structurel

La garde résout **A** (exclure les sous-concepts des frères) mais casse **B** (inclure les sous-concepts du slug lui-même) quand le slug n'a pas de clé exacte dans prefix_map.

| Slug | Exact match? | Guard fonctionne? | Résultat |
|------|-------------|-------------------|----------|
| vendeur | Non | ✅ Oui | 3 candidats, 0 vendeur_autos_* |
| vendeur_autos | Non | ❌ Non | 0 candidats (cassé) |
| vendeur_quincaillerie | Non | ❌ Non | 0 candidats (cassé) |
| peintre | Non | ✅ Oui | 3 candidats, 0 peintre_indl_* |
| administrateur | Oui | N/A (exact match) | OK |

## Solution proposée

Pour les slugs sans clé exacte, utiliser le guard **uniquement sur les clés qui ne commencent PAS par `slug_`** :

```python
for key, node_list in prefix_map.items():
    if not key.startswith(slug + "_"):
        continue
    # Guard: only exclude keys matching OTHER corpus slug prefixes
    # But skip the check if the key's "stem" (after slug_) is itself a slug
    stem = key[len(slug) + 1:]  # everything after "slug_"
    is_sibling = any(stem == s or stem.startswith(s + "_") for s in corpus_slugs if s != slug)
    if not is_sibling:
        nodes.extend(node_list)
```

Pour `vendeur_autos_representant_en_vente_automobile` :
- `slug = "vendeur_autos"`, `stem = "representant_en_vente_automobile"`
- `any(stem == s or stem.startswith(s + "_") for s in corpus_slugs if s != "vendeur_autos")` → False
- ✅ Inclus

Pour `vendeur_autos_representant...` quand `slug = "vendeur"` :
- `stem = "autos_representant_en_vente_automobile"`
- `any(stem == s or stem.startswith(s + "_") for s in corpus_slugs if s != "vendeur")` → False (car "autos_representant..." n'est pas un slug)
- ❌ Inclus (mais ne devrait pas l'être)

**Le conflit persiste** : la garde ne peut pas distinguer "sous-concept du slug" de "sous-concept d'un frère" quand le frère n'a pas de clé exacte.

## Alternative : pas de guard pour les slugs sans clé exacte

```python
if slug in prefix_map:
    nodes = list(prefix_map[slug])
    # Guard only applies to exact-match slugs (they have their own nodes)
    for key, node_list in prefix_map.items():
        if key.startswith(slug + "_"):
            is_sibling = any(key.startswith(s + "_") for s in corpus_slugs if s != slug)
            if not is_sibling:
                nodes.extend(node_list)
else:
    # No exact match: include ALL prefix matches (no guard)
    # Risk: may include sibling sub-concepts, but at least won't lose data
    for key, node_list in prefix_map.items():
        if key.startswith(slug + "_"):
            nodes.extend(node_list)
```

**Résultat** :
- `vendeur` (pas d'exact match) : 7 candidats (dont vendeur_autos_*, vendeur_quincaillerie_*)
- `vendeur_autos` (pas d'exact match) : 2 candidats (ses propres noeuds)
- `administrateur` (exact match) : guard actif, pas de contamination

**Inconvénient** : `vendeur` inclut encore les sous-concepts de ses frères.

## Recommandation

L'alternative "pas de guard pour les slugs sans clé exacte" est le meilleur compromis :
- Aucun slug ne devient manquant (pas de regression)
- Les slugs avec exact match (majorité) ont le guard actif
- Les slugs sans exact match (minorité : vendeur, peintre, etc.) incluent potentiellement des frères, mais c'est préférable à perdre des données

Le guard est documenté comme **best-effort** : il réduit la contamination mais ne l'élimine pas pour les slugs hiérarchiques sans clé exacte.
