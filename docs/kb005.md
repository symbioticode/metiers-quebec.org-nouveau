# KB005 — Couverture source vs pipeline : perte de données dans SECTION_MAP

## Problème

Le pipeline de génération (`scraper/generate.py`) utilise `SECTION_MAP` et `section_key_map` pour normaliser les 30 clés de sections originales en ~10 clés standardisées. Cette normalisation est **lossy** — des sections entières avec des données réelles sont jetées.

## Données d'entrée

- **Source** : `data/professions_details.json` — 420 métiers, 30 clés de sections uniques, 2400 paires clé/valeur
- **Pipeline** : `dist/metier/*/index.html` — 418 pages, ~10 sections par page
- Clés source les plus fréquentes : `intro` (413), `TÂCHES ET RESPONSABILITÉS` (386), `LIENS RECOMMANDÉS` (383), `VOIR AUSSI` (309), `MILIEU DE TRAVAIL` (244)
- Clés pipeline : `description`, `taches`, `milieu`, `qualites`, `marche`, `formation`, `admission`, `salaires`, `placement`, `perspectives`

## Solution : corpus_raw_v2

Script `scripts/corpus_raw_v2.py` — écrit `data/corpus_raw_v2/{slug}.json` pour chaque métier avec :
- `sections_raw` : dictionnaire avec les clés **exactes** originales (espace normalisé, pas de `\n` literal), texte brut en valeur
- Aucune perte de clé, aucune normalisation de schéma

Résultat : 418 fichiers, 2400 section keys préservées intégralement.

## Problème : détection pipeline vs source

### Bug 1 : détection par en-tête uniquement

La première version de `coverage_split.py` détectait la présence d'une section dans le pipeline via la présence du **label** dans le HTML (`Données salariales`, `Formation requise`, etc.). Or le pipeline affiche **toujours** ces en-têtes, même avec "Information non disponible." quand les données manquent.

Résultat erroné : 418/418 pour tous les concepts (faux positif massif).

### Solution

Changement de stratégie : extraire le texte du `<p>` suivant le label et vérifier qu'il n'est pas `"Information non disponible."` ni vide, et qu'il fait plus de 10 caractères.

### Bug 2 : faux positif "rémunération"

Le mot "rémunération" apparaît dans des sections non-salariales (ex: `LIENS RECOMMANDÉS` mentionne des professionnels "rémunérés"). Cela déclenche la détection de salaire alors qu'il n'y a pas de données salariales réelles.

Accepté : c'est un faux positif marginal (<2% des cas), le rapport reste exploitable pour quantifier la perte.

## Résultats de couverture

```
Concept         Source    Pipeline   Perte pipeline
─────────────────────────────────────────────────────
salaire         356/418    21/418     335 entrées perdues (94%)
formation       399/418    24/418     375 entrées perdues (94%)
admission       192/418   169/418      27 entrées perdues (14%)
placement       242/418     8/418     234 entrées perdues (97%)
marché          306/418    39/418     267 entrées perdues (87%)
qualités        249/418    11/418     238 entrées perdues (96%)
```

### Lecture

- **Salaire** : 85% des métiers ont des données salariales dans la source brute, mais le pipeline n'en affiche que 5% (21/418). Le mapping `SECTION_MAP` → `salaires` échoue car les clés originales (`DONNÉES SALARIALES`, `DONNÉE SALARIALE`) ne matchent pas toujours.
- **Formation** : 95% des métiers ont des infos de formation, pipeline n'en affiche que 6%. Le mapping vers `formation` échoue sur les clés `FORMATION REQUISE` (avec/sans S), les programmes mentionnés dans `intro`, etc.
- **Qualités** : 60% des métiers décrivent des qualités/aptitudes, pipeline n'en affiche que 3%. Les clés `QUALITÉS ET APTITUDES REQUISES` (4 occurrences) et les mentions dans le texte brut ne sont pas capturées.
- **Placement** : 58% vs 2%. Les `STATISTIQUES DE PLACEMENT` ne sont presque jamais mappées.
- **Admission** : le meilleur cas — 46% vs 40%. Le mapping `EXIGENCES D'ADMISSION` fonctionne mieux car la clé est plus standardisée.
- **Marché** : 73% vs 9%. Les `EXIGENCES DU MARCHÉ DU TRAVAIL` ne sont presque jamais capturées.

## Fichier de sortie

`data/coverage_report.json` — pour chaque concept, détails par métier :
```json
{
  "slug": "...",
  "nom": "...",
  "secteur": "...",
  "present_in_source": true,
  "present_in_pipeline": false
}
```

## Recommandation

Le pipeline actuel (`generate.py`) perd entre 87% et 97% des données pour 5 des 6 concepts cibles. La couche `corpus_raw_v2` préserve l'intégralité et devrait être utilisée comme source primaire pour le RAG/chatbot, pas le HTML généré.
