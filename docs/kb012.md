# kb012 — Couche atomique : implémentation et validation

## Contexte

kb008-bis définit la nomenclature (FAIT/GARDE/ORIGINE/CODE) et le format
atomique. Ce document couvre l'**implémentation concrète** : les GARDES
codées dans `scripts/gardes.py`, l'algorithme de résolution de conflits,
les résultats de validation, et l'état d'avancement des phases.

## Les cinq GARDES

Toutes codées en Python stdlib uniquement dans `scripts/gardes.py`.
Vérifiables indépendamment via `scripts/test_gardes.py` (17 tests)
ou en bout-en-bout via `scripts/smoke_test_atomique.py`.

### GARDE-PARTITION

Deux fiches ne peuvent pas porter le même `code`. Si `code` est un
identifiant canonique (CNP), chaque métier n'apparaît qu'une seule fois
dans `data/atomic/`.

```
Vérifie : aucune collision de code entre fiches
Résultat : 413 conformes, 0 violations
```

### GARDE-AUTOJUGEMENT

Une source ne fixe jamais elle-même sa propre `confiance` comme critère
de résolution. Si deux sources divergent sur un même `champ` pour le
même `code`, c'est l'algorithme de résolution de conflit (pas l'un des
deux ingesteurs) qui tranche. L'ingestion ne fait que proposer un FAIT
avec son ORIGINE.

```
Source : PCCD INV-01 / RKA-INV-01
Vérifie : aucune source ne se déclare elle-même "haute" par rapport
          à une autre source dans la même fiche
Résultat : 1002 conformes (tous les FAITS), 0 violations
```

### GARDE-NON-RÉÉCRITURE

La `valeur` d'un FAIT ne doit jamais s'écarter sémantiquement de la
source brute (`corpus_raw_v2` ou `sources_raw`). Pas de paraphrase,
pas d'interprétation silencieuse lors de l'ingestion.

`garde_non_reecriture` accepte un paramètre `source_attendue` : les
FAITS dont l'`origine.source` diffère de `source_attendue` sont
comptabilisés en `non_verifiable` (pas en violation). Ceci permet
d'ingérer des FAITS depuis plusieurs sources sans fausser les
statistiques de conformité.

```
Source : PCCD INV-08 / RKA-INV-08
Paramètre : source_attendue (ex. "metiers_quebec")
Résultat : 996 conformes, 0 violations, 6 non_verifiable
           (les 6 FAITS imt_en_ligne, source != metiers_quebec)
```

### GARDE-IRRÉVERSIBILITÉ

Jamais d'écrasement en place d'un FAIT existant. Une nouvelle donnée
pour le même `code` + `champ` s'ajoute comme un nouveau FAIT avec sa
propre `date_captee` ; l'ancien reste dans la liste. La GARDE de
résolution de conflit choisit lequel exposer en priorité, mais aucun
FAIT n'est supprimé ou modifié rétroactivement.

```
Source : DUO axiome D4
Vérifie : aucun FAIT n'est écrasé, la liste croît ou reste stable
Résultat : 413 conformes, 0 violations
```

### GARDE-COMPLÉTUDE

Chaque fiche doit déclarer explicitement quels champs sont présents
(`completude.champ: true`) et quels champs sont absents
(`completude.champ: false`). Pas de silence — un champ absent doit
l'être par `false`, pas par omission.

```
Vérifie : les clés de completude correspondent aux champs réellement
          présents dans la liste de faits
Résultat : 413 conformes, 0 violations
```

## Algorithme de résolution de conflits

Script : `scripts/resoudre_conflits.py`

Pour chaque (`code`, `champ`) où plusieurs FAITS de sources différentes
existent :

1. **Hiérarchie de confiance** : `haute` > `moyenne` > `basse`
2. **Si égalité** : retourne `conflit_non_resolu` sans inventer de
   gagnant (GARDE-AUTOJUGEMENT)
3. **Le FAIT retenu** est celui dont la source a la confiance la plus
   haute ; les autres restent dans la liste (GARDE-IRRÉVERSIBILITÉ)

Le test `--test-egalite` génère deux FAITS synthétiques (source_a 50 000$
vs source_b 55 000$, même confiance) et vérifie que le conflit est
correctement identifié comme non résolu.

## État de Phase 3 (conflit multi-source)

### Ce qui fonctionne

- `resoudre_conflits.py` résout correctement les conflits par confiance
- Le test d'égalité de confiance retourne `conflit_non_resolu` sans
  inventer de gagnant
- `garde_non_reecriture` comptabilise les FAITS imt_en_ligne en
  `non_verifiable` (source != metiers_quebec)
- 6 FAITS réels imt_en_ligne ingérés avec succès (confiance=haute,
  methode=manuel, reference=URL quebec.ca)

### Ce qui manque

- `metiers_quebec` n'a pas de salaires structurés dans `corpus_raw_v2`.
  Le texte brut contient des données salariales (ex: architecte 68 300$
  annuel, plombier 22,68$/hre) mais `ingest_corpus_raw_v2.py` ne les
  extrait pas — c'est un gap d'extraction documenté, pas une valeur à
  inventer.
- Sans FAIT `metiers_quebec/salaire_median` réel, il n'y a pas de
  conflit inter-sources à résoudre.

### Pour compléter Phase 3

1. Soit une troisième source avec des salaires structurés pour les
   mêmes métiers (ex: ISQ, Guide MESS)
2. Soit extraire les salaires du texte libre de `corpus_raw_v2`
   (reconnaissance de montants horaires/annuels) — travail d'extraction
   distinct du format atomique

| Métier | CNP | Salaire | Source |

| Plombier | 72301 | 22,68 $/h | donneesquebec.ca |
| Vendeur | 66211 | 18,00 $/h | donneesquebec.ca |

Chaque FAIT porte :
- `origine.confiance: "haute"` (donnée gouvernementale)
- `origine.methode: "manuel"` (extraction humaine, pas scraping)
- `origine.reference`: URL complète de la page source
- `code_provisoire: true` (mapping slug→CNP non encore validé)

## Gaps d'extraction documentés

Le script `ingest_corpus_raw_v2.py` n'extrait pas les salaires du texte
libre présent dans `corpus_raw_v2`. Exemples dans le brut :

- `architecte.json` : « Salaire annuel moyen : 68 300 $ »
- `plombier.json` : « Taux horaire moyen : 22,68 $ »
- `comptabilite.json` : « Taux horaire moyen : 20,00 $ »

Ces valeurs ne sont pas injectées artificiellement — elles restent dans
le brut en attente d'un extracteur dédié.

## Mapping slug → CNP

**Option B** retenue : le slug du fichier dans `data/atomic/` est utilisé
comme `code` de manière provisoire (`code_provisoire: true`). Le vrai
CNP 2021 sera assigné lors d'un mapping formel ultérieur. Cette décision
permet de progresser sur l'ingestion sans bloquer sur la validation
CNP de chaque métier.
