## Fix graph_bridge : avant/après

| Métrique | Avant (exact match) | Après (prefix match) |
|----------|--------------------|-----------------------|
| Slugs mappés | 305/418 (73%) | **416/418 (99%)** |
| Communautés | 29 | 83 |
| Sous-nœuds | 0 | 312 (74 slugs multi-candidats) |
| Absences réelles | 113 (fausses) | **2 (vraies)** |

**Pourquoi l'ancien taux était faux** : le matching par égalité stricte (`node["id"] == "metier_" + slug`) ne trouvait que les nœuds dont l'ID est exactement le slug. Or Graphify crée souvent des sous-nœuds sémantiques (ex: `metier_administrateur_analyste_en_strategies_d_affaires` pour le slug `administrateur`). Ces sous-nœuds existent bien dans le graphe mais n'étaient pas comptés. Le fix par préfixe les capture tous, et garde le nœud le plus court comme "page entière".

Seules 2 absences sont réelles : `commis-comptable` (trait d'union dans l'ID, incompatible) et `controleur_maritimel` (artefact de scraping, 'l' final).

## Problèmes identifiés

1. **Pas de rejet hors-corpus** — Q8 (astronaute) retourne un chunk non pertinent au lieu de "absent". Le seuil de 0.3 est trop bas. Il faudrait un seuil dynamique ou un classifier binaire "in-corpus / out-of-corpus".

2. **Mauvaise priorisation des sections** — Q7 retourne un chunk non pertinent en tête alors que le bon chunk est en 3e position. Le embedding ne discrimine pas assez entre les sections d'un même métier.

3. **Confusion entre métiers similaires** — Q4 confond "technicien en informatique" avec "technicien en bureautique". Les embeddings sont trop proches pour des métiers apparentés.

4. **Pas de signal quand les données sont absentes** — Q2 et Q3 devraient dire "les données salariales/qualités ne sont pas présentes dans le brut pour ce métier" au lieu de retourner un chunk non pertinent.

## Recommandations

1. **Seuil de rejet**: Augmenter à 0.5 + vérifier que le slug du top result correspond à un métier mentionné dans la question
2. **Filtrage par slug**: Si la question mentionne un métier spécifique, filtrer les résultats sur ce slug avant scoring
3. **Pipeline status**: Toujours afficher "present_in_pipeline: true/false" pour chaque chunk retourné
4. **Graph enrichment**: Utiliser les voisins du graphe pour les questions de type "quels métiers proches de X"
