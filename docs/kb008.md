# kb007 — Métacognition et détection de bugs : une architecture formelle

## Contexte

Lors de la révision de `scripts/graph_bridge.py` (après le Jour 2), une collision de préfixe a été identifiée : un slug `vendeur` n'ayant pas de nœud exact dans le graphe provoque un balayage par préfixe qui attrape involontairement les nœuds de métiers distincts (`vendeur_autos`, `vendeur_mecanique`).

Cette KB documente non pas juste le bug, mais **le processus de détection lui-même**, et formalise la couche commune qui permet à une machine et un humain de collaborer sur une même correction logique sans partager le même substrat de représentation.

## Hypothèse : une couche d'invariants formels indépendante du substrat

Quand Claude (IA, activations vectorielles), Big Pickle (correcteur Python) et Andrei (humain, concept mental) collaborent sur la même erreur, ils ne partagent pas :
- La même mémoire
- La même implémentation cognitive
- Le même « espace mental » ou « espace vectoriel »

Mais ils partagent **accès à une couche abstraite** : la structure logique de la contradiction elle-même.

Cette couche existe **indépendamment du substrat**. C'est ce qu'on formalise ici.

## Architecture formelle : trois couches

### Couche 1 — Les données (D)

**Définition** : un ensemble fini d'entités et leurs relations.

Dans notre cas :
```
D = {
  corpus_slugs: {vendeur, vendeur_autos, vendeur_mecanique, ...},
  graph_nodes: {metier_vendeur_vendeur, metier_vendeur_autos_*, ...},
  belongs_to(node, slug): une relation binaire définissant quel nœud
                           appartient à quel concept source
}
```

**Propriété invariante** : chaque slug du corpus a une **partition disjointe** de nœuds qui le représentent.

```
Pour tout (S1, S2) dans corpus_slugs :
  Si S1 ≠ S2, alors nodes(S1) ∩ nodes(S2) = ∅
```

Cette propriété est définie par l'intention du corpus : chaque métier est une entité distincte.

### Couche 2 — La logique (L)

**Définition** : un ensemble de règles de transformation et de test sur D.

Dans notre cas, `find_candidates_for_slug()` implémente :

```
L = {
  Rule 1 (exact match)  : ∃ node ∈ graph_nodes où node.id = "metier_" + slug
                           → return node
  Rule 2 (prefix scan)  : ∀ node ∈ graph_nodes où node.id commence par
                           "metier_" + slug + "_"  → add à candidates
  Rule 3 (select main)  : parmi candidates, le nœud au plus court ID
                           devient main_node
}
```

La question devient : **« L préserve-t-elle l'invariant de D ? »**

### Couche 3 — Les contradictions (C)

**Définition** : une contradiction est une **violation prouvée d'un invariant**.

```
C = cas où L(D) produit :
  ∃ (S1, S2) ∈ corpus_slugs, S1 ≠ S2 :
    ∃ node partagé entre nodes_L(S1) et nodes_L(S2)
```

**Cas concret** : Rule 2 appliquée à `vendeur` sans garde de collision fait que
`nodes_L(vendeur)` et `nodes_L(vendeur_autos)` se recoupent. C'est une
**contradiction** — une assertion fausse sur la structure du corpus.

## Processus de détection : navigation à travers D, L, C

### Phase 1 — Lecture de L (le code)

```python
if key.startswith(slug + "_"):  # ← Rule 2
    nodes.extend(node_list)
```

**Acte cognitif** : extraire la structure logique du code syntaxique — transformer
`if key.startswith(...)` en règle formelle explicite, et la maintenir active
pendant l'analyse.

### Phase 2 — Consultation de D (les données réelles)

Question posée : « Existe-t-il un cas où Rule 2 viole l'invariant de partition ? »

```
∃ slug1, slug2 ∈ corpus_slugs :
  slug1 ≠ slug2 AND slug2.startswith(slug1 + "_")
?
Réponse : OUI — (vendeur, vendeur_autos), (peintre, peintre_indl)
```

Cette vérification peut être manuelle, algorithmique, ou vectorielle — le
**contenu logique de la vérification** est identique quel que soit le substrat.

### Phase 3 — Détection de C (la contradiction), preuve formelle

```
Soit slug1 = "vendeur", slug2 = "vendeur_autos"

Appliquer L à "vendeur" :
  "vendeur_autos" commence par "vendeur_"
  → nodes_L(vendeur) inclut metier_vendeur_autos_*

Appliquer L à "vendeur_autos" :
  → nodes_L(vendeur_autos) inclut aussi metier_vendeur_autos_*

Donc : nodes_L(vendeur) ∩ nodes_L(vendeur_autos) ≠ ∅
→ violation de l'invariant, C est prouvée.
```

### Phase 4 — Correction via modification de L

```
Rule 2 (corrigé) :
  ∀ node où node.id commence par "metier_" + slug + "_"
    AND key NOT IN corpus_slugs   ← nouvelle garde
    → add à candidates
```

La garde exclut tout nœud dont la clé correspond elle-même à un autre slug
réel du corpus. L'invariant est restauré, et la correction est **dérivée de
L**, pas un patch arbitraire.

## Métacognition : le processus qui comprend un processus

**Définition** : la métacognition est l'application de la logique formelle à
la logique elle-même — une **métalogique**.

Ce qui s'est passé :
1. Extraction de L à partir du code (Phase 1)
2. Test de L contre D pour vérifier un invariant (Phase 2-3)
3. Formulation de la correction **en termes de L** (une garde logique), pas
   comme un patch ad hoc

C'est métacognitif parce que **le même outil (invariants, règles, gardes) a
servi à la fois à construire le code original et à le critiquer**. L'outil
d'analyse et l'objet analysé appartiennent au même langage formel.

## Ce qui reste : les invariants, indépendants du substrat

Claude (vecteurs), Big Pickle (exécution Python) et Andrei (concept mental)
ne partagent aucun substrat commun. Mais tous trois accèdent à la même
couche abstraite :

```
L'invariant  : partition disjointe des slugs du corpus par nœud de graphe
La violation : deux slugs distincts partagent un nœud
La correction: une garde qui restaure la partition
```

### Formalisation

Soit S_data, S_logic, S_mind_humain, S_mind_IA les substrats respectifs des
données, du code, de la pensée humaine, de l'activation vectorielle IA.

Soit **I** l'ensemble des invariants (propriétés logiques qu'on veut préserver).

```
I appartient à un Langage Formel Abstrait (LFA), indépendant de tout substrat.

Chaque acteur A accède à I via son propre S_A,
mais I lui-même ne réside dans aucun S_A — il existe dans LFA.
```

**L'invariant est le terrain commun.** C'est ce qui permet à un humain, une IA
et un programme exécuté de converger sur « la même erreur », sans jamais
partager de représentation interne identique.

## Outils pour adresser n'importe quelle connaissance (logique + code)

1. **Données (D)** — identifier les entités, formaliser leurs relations,
   documenter l'invariant attendu.
2. **Logique (L)** — extraire les règles du code, les exprimer formellement,
   tester « L préserve-t-elle l'invariant sur D ? »
3. **Contradiction (C)** — chercher les cas où D + L violent l'invariant,
   les prouver formellement, les documenter avec des exemples concrets.
4. **Correction** — modifier L pour restaurer l'invariant ; formuler la
   modification en termes logiques (une garde, une règle), pas en patch
   syntaxique isolé.
5. **Métacognition** — appliquer ce même processus (D, L, I) à la critique
   elle-même, ce qui permet une collaboration multi-acteurs (humain + IA)
   sans dépendre d'un substrat partagé.

## Validation

Ce cadre s'applique à deux instances du **même invariant formel** dans ce
projet :
- KB002 — violation de mapping (`SECTION_MAP` vs clés réelles du brut)
- KB007 — violation de partition (`corpus_slugs` vs préfixes de nœuds graphe)

```
∀ conteneur, ∀ contenu ⊂ conteneur :
  structure(conteneur) ≠ structure(contenu) est possible
  → invariant permanent : toujours valider le contenu contre la structure
    réelle, jamais présumer qu'elles coïncident.
```

## Implication pour le projet

Avec ce cadre (D, L, I, C), le projet peut :
1. Formaliser les invariants attendus (qu'est-ce qu'une migration fidèle
   du corpus de Dany Savard ?)
2. Écrire des tests qui vérifient I automatiquement
3. Collaborer sans confusion de substrat (un invariant formel peut être
   vérifié par du code, ou relu par un humain, avec le même résultat)
4. Généraliser les corrections — reconnaître la même violation de
   partition si elle réapparaît ailleurs dans le pipeline
